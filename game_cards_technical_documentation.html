1. Resumen funcional

El módulo de cartas es un minijuego frontend con catálogo maestro en MySQL. El servidor expone cartas activas desde fact_game_card_collection mediante /api/game_cards.php. El navegador guarda el progreso del jugador en localStorage: cartas obtenidas, copias, Mnemones, sobres comprados, objetos rituales, cartas rememorando y recompensas gratuitas.

La experiencia tiene seis superficies principales:

  • Sobres/gacha: abre sobres, genera copias con estadísticas aleatorias y reproduce animaciones/sonidos.
  • Tienda: permite comprar sobres y objetos rituales con Mnemones.
  • Colección: muestra álbum con huecos de todas las cartas y tabla de cartas obtenidas, filtrable por tipo, rareza y estado.
  • Progresión: permite evolucionar rareza y mejorar calidad/atributos consumiendo cartas, Mnemones y, en rarezas altas, objetos rituales.
  • Recuerdos: asigna hasta 5 cartas a rememorar para generar Mnemones pasivos.
  • Combate: permite preparar equipos de 5 copias concretas y jugar entrenamientos contra rivales generados desde el catálogo.

El progreso del usuario no se sincroniza con la base de datos. Si se borra el navegador o se limpia localStorage, se pierde salvo que el usuario haya exportado el JSON.

fact_game_card_collection es el catálogo maestro del juego. Cada fila representa una carta coleccionable, no una copia del jugador.

Exclusión de crónicas

El seed respeta dim_web_configuration.config_name = 'exclude_chronicles'. Es una lista CSV de IDs de crónica. Se aplica a personajes, crónicas, temporadas y episodios. Además, las cartas ya existentes de crónicas excluidas se marcan como is_active = 0.

4. Seed y generación del catálogo

El seed se ejecuta por CLI:

php app/tools/seed_game_cards.php

Para borrar el catálogo y regenerar desde cero:

php app/tools/seed_game_cards.php --reset

--reset hace DELETE FROM fact_game_card_collection, reinicia el autoincrement y vuelve a insertar todas las cartas generables.

Qué hace el seed

  1. Incluye game_cards_catalog.php.
  2. Ejecuta hg_gc_schema_sql() para crear/actualizar la tabla.
  3. Lee exclude_chronicles.
  4. Genera arrays de cartas desde tablas fuente.
  5. Inserta con ON DUPLICATE KEY UPDATE usando la clave única (source_type, source_table, source_id).
  6. Activa cartas insertadas/actualizadas con is_active = 1.

Criterios de rareza principales

  • Personajes: combinan puntuación por rango, tipo pj y número de relaciones.
  • Niveles/coste: objetos, dones, ritos, tótems y disciplinas usan niveles o coste para mapear rareza.
  • Formas: suman bonos positivos de Fuerza, Destreza y Resistencia, más regeneración y hpregen limitado. Escala actual: >=11 mythic, >=8 legendary, >=6 epic, >=4 rare, >=2 unusual, resto common.
  • Crónicas/sistemas/linaje: tienen criterios simples y conservadores para evitar sobrepoblar rarezas altas.

5. API del catálogo

/api/game_cards.php devuelve JSON con todas las cartas activas. Solo acepta GET y HEAD.

{
  "success": true,
  "cards": [
    {
      "card_id": 1,
      "source_type": "character",
      "source_id": 23,
      "card_name": "Nombre",
      "card_text": "Extracto",
      "card_image_url": "/img/...",
      "card_url": "/characters/slug",
      "card_rarity": "rare",
      "hp_min": 50,
      "hp_max": 85,
      "atk_min": 50,
      "atk_max": 85,
      "def_min": 50,
      "def_max": 85
    }
  ]
}

La API valida cada fila con hg_gc_allowed_source() y hg_gc_valid_rarity(). También normaliza imágenes con hg_gc_normalize_image_url(). Para personajes, si la imagen de carta está vacía o es fallback, intenta usar fact_characters.image_url.

Visibilidad de sobres

  • Los sobres solo se muestran en la pantalla de apertura si el jugador tiene stock en collection.packInventory.
  • Si no hay unidades de ningún tipo, se muestra un mensaje indicando que se pueden comprar más en tienda o probar suerte en Mazmorras.
  • El inventario de sobres está limitado a 99 unidades por tipo. La tienda bloquea compras o reclamaciones que superarían ese límite.
  • Abrir todos consume todo el stock no admin y solo renderiza las últimas 5 cartas obtenidas para evitar bloquear el navegador.
  • En modo admin todos los sobres se consideran disponibles y se muestran con stock Admin.

Funcionamiento de apertura

  1. El usuario pulsa un sobre.
  2. openPack(packKind) valida que el pack exista y que haya stock.
  3. Selecciona rareza con pesos de PACK_RARITY_WEIGHTS.
  4. Filtra cartas compatibles con cardAllowedForPack().
  5. Elige cartas aleatorias y genera estadísticas dentro de rangos.
  6. Añade copias a ownedCards y guarda en localStorage.
  7. Renderiza reveal, resultados, contadores y, cuando corresponde, colección.

Recompensas gratuitas

  • Hasta 3 sobres mnemónicos gratis al día reclamados desde tienda.
  • El cupo diario se guarda en localStorage por fecha local.
  • Los Mnemones ya no se generan gratis por temporizador.
  • La generación pasiva de Mnemones depende de cartas asignadas a recordar.

La web no recibe eventos push del servidor. El cliente resetea el cupo gratis al detectar un cambio de fecha local.

Optimizaciones de tienda

  • Las compras usan render ligero: actualizan moneda, tienda y stock sin recalcular colección, recuerdos ni combate.
  • addPack() acepta modo silencioso para evitar doble guardado/doble render cuando lo llama buyPack().
  • Los contadores de Mnemones usan separador español mediante toLocaleString('es-ES').

8. Colección

La colección usa el catálogo completo para mostrar huecos. Esto significa que una carta no obtenida aparece en álbum como hueco, con rareza y posición, pero sin imagen ni nombre real visible.

Álbum

  • Muestra todas las cartas activas del catálogo, obtenidas o no.
  • Se filtra por categoría: todos, personaje, sistema, tribu, auspicio, forma, etc.
  • Cada filtro muestra icono, etiqueta y contador obtenidas / total.
  • En desktop usa una grilla de 4 cartas de ancho y comparte ancho máximo con sobres, tienda y combate.
  • Al pulsar una carta obtenida abre modal y reproduce /sounds/ui/card.ogg.

Tabla

  • Muestra solo cartas obtenidas.
  • Ordena por mejor puntuación total.
  • Incluye rareza, calidad, nombre, categoría, ID, PS, ATQ, DEF, total, copias y fecha.
  • En móvil se reduce a columnas compactas.

Modal de copias

  • La carta principal se muestra completa y las variantes obtenidas aparecen a la derecha también cuando solo existe una copia.
  • Cada variante muestra solo calidad y total de puntos; PS, ATQ y DEF se reservan para la carta completa y el contexto de combate.
  • Los comandos de copia usan iconos PNG de /public/img/ui/card_game_icons: vender, mejorar y evolucionar, con title para hover.
  • El comando Recuerdos no aparece en las cartas de colección; esa gestión se hace exclusivamente desde la pestaña Recuerdos.

Paginación y vista

  • Modos: album y table.
  • Tamaños: 10, 20 o 50 cartas por página.
  • La preferencia se guarda en localStorage.

Exportar, importar y borrar

  • Exportar JSON: descarga la colección local completa.
  • Importar JSON: reemplaza la colección local por un JSON válido.
  • Borrar colección local: reinicia progreso del navegador.

9. Progresión de cartas

La progresión opera sobre copias concretas, no sobre la carta base. Por eso una misma carta puede tener varias copias con rareza, calidad y atributos distintos. La nomenclatura visible queda separada así: Evolucionar sube rareza y Mejorar sube calidad/atributos.

Evolucionar rareza

  • El flujo visible se llama Evolucionar y se abre desde el modal de variantes de una carta.
  • La copia objetivo sube a la siguiente rareza manteniendo su instanceId.
  • Se requieren 5 puntos de progreso de sacrificio. Las copias sacrificadas deben ser de rareza igual o superior a la copia objetivo, nunca inferior.
  • Las copias sacrificadas deben tener al menos 50% de calidad.
  • Las copias rememorando o usadas en equipos no aparecen como sacrificios.
  • Las rarezas superiores aportan multiplicadores: misma rareza x1, +1 x1.2, +2 x1.5, +3 o más x2.
  • Al evolucionar, retuneCopyStatsForRarity() reescala PS/ATQ/DEF al rango de la nueva rareza conservando la posición porcentual de la copia.
  • El modal muestra barras de calidad objetivo y progreso de rareza, filtros de sacrificios y una caja de costes antes de confirmar.

Mejorar calidad

  • El flujo visible se llama Mejorar y aumenta calidad/atributos hasta un máximo de 100%.
  • Solo acepta sacrificios de la misma rareza efectiva que la copia objetivo.
  • Se pueden usar hasta 5 sacrificios. La ganancia depende de la calidad del sacrificio y disminuye a medida que la copia objetivo se acerca al 100%.
  • Al confirmar, applyQualityToCopyStats() recalcula PS, ATQ y DEF según la rareza efectiva y la nueva calidad.
  • La selección de sacrificios comparte el patrón de Evolucionar, pero bloquea rarezas distintas para evitar convertir copias en progreso de rareza.

Tanto Mejorar como Evolucionar cuestan Mnemones según rareza efectiva y calidad actual. La calidad entra como factor entre 0.01 y 1.

Restricciones de seguridad

  • La copia objetivo nunca puede aparecer entre los sacrificios.
  • Las copias en equipos de combate se excluyen para no dejar equipos con referencias rotas.
  • Las copias rememorando se excluyen porque están bloqueadas por el sistema de Recuerdos.
  • Si una acción consume copias, los equipos se limpian después con las mismas rutinas de validación usadas por combate.

10. Recuerdos

Recuerdos sustituye el antiguo concepto de trabajar. Las cartas que están rememorando generan Mnemones, fragmentos de Mnemógeno, de forma pasiva.

  • Máximo de 5 cartas rememorando a la vez.
  • Cada carta debe permanecer al menos 24 horas antes de poder retirarse.
  • Se puede reclamar lo acumulado sin retirar cartas.
  • Al retirar una carta, sus ganancias pendientes pasan a workPendingRewards y quedan para reclamar.
  • Las cartas rememorando no pueden venderse, sacrificarse, evolucionarse, mejorarse ni usarse en combate.
  • La producción usa rareza efectiva, calidad y total de PS + ATQ + DEF. La función principal es workRatePerMinute(copy, card).

La vista de Recuerdos muestra cartas completas en una parrilla de 3 arriba y 2 abajo en desktop, y una distribución adaptada en móvil. Bajo cada carta se muestra lo que genera por minuto y el botón Retirar; la reclamación de ganancias acumuladas queda en el botón superior de la sección.

La vista de Recuerdos se renderiza solo cuando está visible en móvil. Al salir del panel, las listas pesadas se vacían del DOM para liberar memoria.

11. Combate

El combate es una superficie frontend local, disponible en /games/card-game/combat y en el panel combat de la vista móvil. Usa el mismo catálogo y las copias guardadas en hg_card_collection_v1; no persiste resultados en servidor.

Rutas y navegación

  • request_router.php resuelve /games/card-game/combat como p=game_cards_combat.
  • body_work.php registra título y metadatos para game_cards_combat.
  • Las pestañas de sobres, tienda, colección, recuerdos, combate e información comparten navegación en desktop y móvil.

Preparación de equipo

  • El jugador puede guardar hasta 5 equipos, cada uno con 5 copias concretas identificadas por instanceId.
  • El selector de cartas usa las copias obtenidas, no solo cartas únicas. Puede filtrar por rareza, tipo de carta y por copias todavía no elegidas.
  • El selector muestra valor total de equipo y, al cambiar cartas, solo enseña PS actual / total, ATQ y DEF para reducir ruido visual.
  • autoBuildCombatTeam() crea un equipo rápido con las 5 mejores copias disponibles por PS + ATQ + DEF, excluyendo copias rememorando.
  • El perfil local guarda nombre del jugador y carta favorita en hg_card_combat_profile_v1.
  • Si una copia desaparece de la colección, validDraftTeam() limpia el equipo antes de guardarlo/renderizarlo.

Entrenamiento

  1. startTrainingCombat() valida que el equipo activo tenga 5 copias válidas. Si no las tiene, abre una confirmación para crear un equipo rápido, seleccionarlo y empezar.
  2. Genera 5 enemigos desde el catálogo, evitando repetir card_id cuando sea posible.
  3. La dificultad seleccionada define pesos de rareza para elegir enemigos y multiplicador de recompensa.
  4. Las acciones disponibles son atacar, defender, cambiar carta y huir. Huir finaliza el combate sin recompensa ni pérdida de cartas.
  5. El HUD muestra carta activa, PS, ATQ, DEF efectivo, escudos y registro de combate.

Dificultades y recompensa

Aprendizapprentice15-25 Mnemones

La fórmula de victoria es 5 * random(1,5) * multiplicador, normalizada con clampInt(). El multiplicador se captura al iniciar el combate en state.combat.rewardMultiplier, para que cambiar el selector durante una partida no altere la recompensa del rival ya generado.

Reglas de combate

  • La vida inicial de cada unidad sale de la copia: copy.hp. ATQ y DEF también usan la copia concreta.
  • Los escudos dependen de rareza mediante rarityShieldCount(): común tiene 1, inusual 2, raro 3, épico 4, legendario 5 y mítico 6.
  • Defender consume 1 escudo, activa defensa aumentada y cura aproximadamente un tercio de los PS máximos.
  • El daño base compara ATQ contra DEF efectivo y añade un extra aleatorio ajustado por diferencia de rareza.
  • Al caer una carta, advanceDefeatedSide() elige automáticamente la siguiente copia viva. Si no quedan copias vivas, marca victoria o derrota.

Final y animaciones

  • Al ganar se renderiza un overlay dentro de .hg-combat-screen con ¡Superaste el entrenamiento!, recompensa obtenida y botón para iniciar otro combate.
  • Al perder se renderiza ¡Te han derrotado!, aviso de que no se pierden cartas y el mismo botón de reinicio.
  • Los tiempos de animación están centralizados en constantes JS: COMBAT_ATTACK_MS, COMBAT_DEFEND_MS, COMBAT_DEFEAT_MS, COMBAT_ENTRY_MS y COMBAT_TURN_GAP_MS.
  • La cadena de derrota se mantiene sincronizada: ataque/defensa, derrota completa, render de nueva carta, entrada completa y desbloqueo de acciones.

Audio y rendimiento

  • Los sonidos de combate usan /public/sounds/ui/attack1.ogg, attack2.ogg, hit1.ogg, hit2.ogg, heal.ogg y card_defeat.ogg.
  • El ataque y el daño varían ligeramente playbackRate para evitar repetición. El golpe se retrasa con COMBAT_HIT_SOUND_DELAY_MS para alinearlo con la animación.
  • Los sonidos se cachean y reutilizan desde uiSoundCache. No se clonan nodos de audio para evitar picos de memoria en Firefox y móvil.
  • renderCombatUnit() no reconstruye la carta activa si el instanceId no cambia; así se evita el parpadeo de avatares durante animaciones.

12. UI, modales y responsive

Confirmaciones propias

Las acciones destructivas ya no usan window.confirm(). confirmGameAction() crea un modal flotante .hg-confirm-modal, con cierre por fondo, botón cancelar y tecla Escape.

  • Se usa para desintegrar copias legendarias/míticas, desintegrar duplicadas, desintegrar todas, vender por rareza, borrar colección local y crear equipo rápido al iniciar combate.
  • Las funciones afectadas aceptan un parámetro confirmed para reentrar tras aceptar sin volver a abrir el modal.
  • El modal tiene variante móvil .hg-confirm-modal--mobile y comparte estética con sobres, combate y cartas.

Mobile y viewport

  • .hg-card-mobile-body, .hg-cards--mobile, .hg-mobile-panels y paneles internos fijan ancho máximo al viewport y ocultan overflow horizontal.
  • La vista móvil standalone usa viewport-fit=cover en game_cards_mobile.php y CSS con env(safe-area-inset-*).
  • Las pestañas móviles usan las mismas secciones que desktop: sobres, tienda, colección, recuerdos, combate e información. En pantallas estrechas reducen padding y tamaño de iconos para evitar scroll horizontal.
  • La navegación principal y los comandos de carta usan iconos PNG de 64x64 en /public/img/ui/card_game_icons, manteniendo texto solo en title o contexto visible cuando aporta significado.
  • Los inputs dentro de .hg-collection-select comparten estilo con selects para soportar el perfil de combate.
  • activeMobilePanel(), isCollectionContext(), isMemoryContext() y isCombatContext() evitan renderizar paneles pesados cuando no están visibles.
  • unloadInactiveMobilePanels() vacía álbum, tabla, recuerdos y preparación de combate al cambiar de sección para liberar memoria del DOM.
  • El sonido de navegación entre pestañas se retiró para mejorar respuesta en móviles con colecciones grandes.

13. Administración

admin_game_cards.php permite revisar y editar cartas del catálogo en fact_game_card_collection. Sirve para correcciones manuales después del seed: nombre, texto, imagen, rareza, stats e is_active.

Si una carta procede de una tabla fuente y se vuelve a ejecutar el seed, puede ser actualizada por ON DUPLICATE KEY UPDATE. Para cambios permanentes conviene modificar la lógica del seed o la tabla origen, no solo la carta generada.

14. Cómo ampliar el módulo

Añadir un nuevo tipo de carta

  1. Elegir source_type estable en inglés, singular y corto.
  2. Añadirlo a hg_gc_source_whitelist() con su tabla origen.
  3. Añadir etiqueta a hg_gc_type_labels().
  4. Añadir URL en hg_gc_card_url().
  5. Añadir fallback en hg_gc_fallback_image_url().
  6. Añadir el valor al ENUM source_type en hg_gc_schema_sql(), tanto en CREATE TABLE como en ALTER TABLE ... MODIFY source_type.
  7. Añadir bloque de seed en hg_gc_seed_rows().
  8. Añadir etiqueta, orden, icono y alias si hace falta en assets/js/game-cards.js.
  9. Si el tipo debe aparecer en un sobre específico, actualizar arrays de tipos y cardAllowedForPack().
  10. Ejecutar php -l y luego php app/tools/seed_game_cards.php.

Añadir cartas individuales

  • Vía tabla origen: crear una fila en la tabla fuente y ejecutar el seed. Es la opción preferida.
  • Vía admin: editar o activar cartas directamente en fact_game_card_collection. Útil para ajustes rápidos.
  • Vía seed manual: ampliar hg_gc_generic_seed_cards() solo para fallback o pruebas; no es el flujo normal.

Añadir un nuevo sobre

  1. Añadir clave a PACK_KINDS.
  2. Añadir precio en PACK_PRICES.
  3. Añadir pesos en PACK_RARITY_WEIGHTS.
  4. Añadir nombre en PACK_LABELS.
  5. Añadir descripción en PACK_CONTENTS, porque la tienda la inyecta desde JS.
  6. Añadir filtro en cardAllowedForPack().
  7. Añadir artículo de tienda con data-shop-pack en desktop y móvil. Los botones x1/x5/x20 se generan desde JS.
  8. Añadir botón de apertura en desktop y móvil. Solo se mostrará si el jugador tiene stock, salvo modo admin.
  9. Verificar que respeta el límite de 99 unidades y que Abrir todos no intenta pintar más de 5 resultados finales.
  10. Añadir clase visual .hg-pack--nuevo en CSS.

Añadir un nuevo objeto ritual

  1. Añadir definición a UPGRADE_MATERIALS: clave, nombre, precio, rareza visual y descripción.
  2. Si bloquea una evolución concreta, añadirlo a RARITY_UPGRADE_MATERIALS.
  3. Añadir artículo de tienda con data-shop-material en desktop y móvil.
  4. Confirmar que compra incrementa collection.materialInventory, descuenta Mnemones y se conserva en export/import.
  5. Crear icono dedicado cuando exista asset; mientras tanto puede usar el icono de rareza asociada.

Añadir una nueva dificultad de combate

  1. Añadir opción en los selects data-combat-difficulty de game_cards_combat_page.php y game_cards_mobile_page.php.
  2. Añadir etiqueta y pesos de rareza en combatDifficultyConfig().
  3. Añadir multiplicador en combatRewardMultiplier().
  4. Probar que el equipo rival se genera, que la recompensa usa el multiplicador capturado al iniciar y que el overlay final se muestra correctamente.

Añadir una nueva colección

El módulo actual solo tiene una colección local por navegador. Para añadir colecciones separadas hay dos caminos:

  • Colección local separada: crear una nueva clave localStorage, un nuevo STORAGE_KEY y una ruta/vista dedicada. Es rápido pero sigue sin sincronización server-side.
  • Colección persistente por usuario: crear tablas nuevas de usuario/copia, endpoints de lectura/escritura y autenticación. Requiere migrar funciones de ownedCards desde localStorage a API.

Si se quiere soportar varias colecciones con el mismo catálogo, la abstracción mínima es parametrizar STORAGE_KEY, textos y filtros iniciales por atributo del contenedor .hg-cards.

  2. Confirmar que ningún inventario de sobres supera 99 unidades.
  3. Reclamar los 3 sobres gratis diarios desde tienda y verificar que no se duplican al cambiar de panel.

  2. Comprobar que los modales de confirmación cierran con cancelar, fondo y Escape.
  3. Comprobar que sobres, tienda, colección, recuerdos, combate e información aparecen como pestañas en desktop y móvil.
  4. Comprobar que los sobres sin stock no aparecen en apertura.
  5. Comprobar que la tienda muestra contenido y precio de cada sobre.
  6. En móvil, cambiar entre colección, recuerdos y combate y confirmar que no quedan listas pesadas de paneles ocultos en pantalla.

Riesgos conocidos

  • El estado local depende de card_id. Si se usa --reset y cambian IDs, colecciones exportadas antiguas pueden quedar desalineadas.
  • El cliente usa fecha local para el cupo diario de sobres gratis. Cambios manuales en fecha del dispositivo pueden alterar el reinicio diario.
  • Los equipos de combate dependen de instanceId. Si una colección se importa, borra o desintegra copias usadas por un equipo, el cliente limpia esos huecos al renderizar/guardar.
  • Las animaciones de combate dependen de que las constantes JS sigan alineadas con las duraciones CSS de @keyframes. Si se cambia una duración visual, actualizar también la constante correspondiente.
  • Los sonidos dependen de permisos del navegador y pueden no reproducirse hasta que exista interacción del usuario. Mantener la caché de audio ligera.
  • Una colección grande sigue ocupando memoria en localStorage; la optimización actual reduce DOM/render, no el tamaño del JSON guardado.
  • Los objetos rituales forman parte de export/import. Si se cambia su clave, hay que migrar materialInventory.
  • La API no pagina; devuelve todo el catálogo activo. Si crece mucho, habrá que añadir paginación o caché.
  • Algunas etiquetas antiguas pueden estar mojibakeadas en archivos existentes; nuevas ediciones deberían guardarse en UTF-8.
